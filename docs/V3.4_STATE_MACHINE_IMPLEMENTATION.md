# V3.4 State Machine Implementation Plan

## Overview
Replace current phase logic with deterministic derived state machine based on PPAP data.

## Current Architecture Analysis

### Current State Management:
1. **Status Field**: `ppap.status` (PPAPStatus enum) - stored in database
2. **Phase Derivation**: `mapStatusToPhase(status)` - derives WorkflowPhase from status
3. **Manual State Updates**: `updatePPAPState()` - single entry point for status changes
4. **Phase Display**: UI uses derived phase from status

### Problems with Current System:
- Status is manually set, not derived from data
- Can have inconsistent states (e.g., status says "complete" but validations incomplete)
- Phase is derived from status, but status itself is manual
- No automatic state transitions based on data changes

## New Architecture: V3.4 Derived State Machine

### Core Principle:
**State is ALWAYS derived from data, NEVER stored**

### Derivation Logic:
```
IF intake confirmations < 3 → INTAKE
ELSE IF preAckCompleteCount < 6 → PRE_ACK_VALIDATION
ELSE IF acknowledged_date is null → READY_FOR_ACK
ELSE IF acknowledged_date exists AND documents incomplete → DOCUMENTATION
ELSE IF documents complete AND submitted_date is null → SUBMISSION_READY
ELSE IF submitted_date exists AND no approval → SUBMITTED
ELSE IF status === 'APPROVED' → APPROVED
ELSE IF status === 'REJECTED' → REJECTED
```

### Data Points Used:
- **Validations**: Count of completed pre-ack validations (target: 6)
- **Intake Confirmations**: First 3 pre-ack validations (target: 3)
- **acknowledged_date**: Timestamp when coordinator acknowledged
- **submitted_date**: Timestamp when package submitted
- **Documents**: Required documents completion status
- **status**: Only for final states (APPROVED/REJECTED)

## Implementation Strategy

### Phase 1: Create Derived State Engine ✓
- Created `derivedStateMachine.ts` with `derivePPAPState()` function
- Implements deterministic logic based on data
- Returns `PPAPStateContext` with state, reason, nextAction

### Phase 2: Integration Points
Need to update these components to use derived state:

1. **PPAPWorkflowWrapper.tsx** - Main workflow container
   - Currently uses: `mapStatusToPhase(ppap.status)`
   - Should use: `derivePPAPState(ppap, validations, documents)`

2. **PPAPHeader.tsx** - Status banner
   - Currently uses: `ppap.status` and derived phase
   - Should use: derived state for display

3. **PPAPOperationsDashboard.tsx** - Dashboard table
   - Currently uses: `ppap.workflow_phase` and `ppap.status`
   - Should use: derived state for filtering/display

4. **PPAPValidationPanelDB.tsx** - Validation panel
   - Currently triggers auto-transitions
   - Should rely on derived state instead

### Phase 3: Remove Manual State Setting
- Keep `updatePPAPState()` for backward compatibility
- But state should be derived, not stored
- Remove auto-transition logic (state derives automatically)

### Phase 4: Update UI Bindings
- Banners show derived state
- Progress indicators use derived state
- Button enablement based on derived state
- No conflicting UI states possible

## Migration Path

### Step 1: Add Derived State (Non-Breaking)
```typescript
// In PPAPWorkflowWrapper
const derivedState = derivePPAPState(ppap, validations, documents);
const selectedPhase = mapDerivedStateToPhase(derivedState.state);
```

### Step 2: Use Derived State for Display
```typescript
// Show derived state in UI
<div>State: {getStateLabel(derivedState.state)}</div>
<div>Reason: {derivedState.reason}</div>
<div>Next: {derivedState.nextAction}</div>
```

### Step 3: Remove Manual Transitions
- Auto-transitions no longer needed
- State updates automatically when data changes
- UI re-renders with new derived state

## Success Criteria

### Functional Requirements:
✓ Cannot display "pre-ack complete" unless 6/6 validations are true
✓ No conflicting UI states (e.g., 0/6 + completed)
✓ State transitions occur automatically when data changes
✓ No manual phase setting anywhere in codebase

### Non-Breaking Requirements:
✓ Existing PPAP creation still works
✓ No database schema changes
✓ UI renders without runtime errors
✓ All existing features continue to function

## Testing Checklist

- [ ] Create new PPAP → shows INTAKE state
- [ ] Complete 3 validations → shows PRE_ACK_VALIDATION
- [ ] Complete 6 validations → shows READY_FOR_ACK
- [ ] Acknowledge → shows DOCUMENTATION
- [ ] Upload documents → shows SUBMISSION_READY
- [ ] Submit → shows SUBMITTED
- [ ] Approve → shows APPROVED
- [ ] Reject → shows REJECTED

## Rollback Plan

If issues arise:
1. Revert to using `mapStatusToPhase(ppap.status)`
2. Keep derived state machine for future use
3. No database changes needed (state not stored)
